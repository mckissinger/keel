# Milestone authoring + verification rules (shared)

The single source of truth for **how a milestone is written and how its "done" is checked.** Two skills point here so the rules never drift: `spec-foundation` *establishes* them once at kickoff (and wires the CI gate); `spec-feature` *applies* them every time it authors a feature's milestones. Neither skill restates these rules — they reference this file.

A **milestone** is the build/verify unit: one branch, one PR, one `verified:` pin, sized for one `implement-milestone` run. A **feature** (the unit `spec-feature` plans) fans out into 1–5 milestones. Everything below is per-milestone.

## 1. Every milestone's "done" is measurable

Write each completion condition so that a verifier subagent (`${CLAUDE_PLUGIN_ROOT}/agents/verifier.md`) can check it against the actual code/outputs, **and** it can be used verbatim as a `implement-milestone` completion condition.

"Auth works" fails this test. "POST /api/login with valid credentials returns 200 and a session cookie; invalid credentials return 401; covered by passing tests in auth.test.ts" passes it.

A done-condition is a per-milestone eval, and the classic grader mistakes produce both false bounces and false pins. Three authoring anti-patterns:

- **Grading implementation paths instead of outcomes.** "Uses a recursive-descent parser" bounces a correct implementation that took another route; assert the observable outcome, never the route to it. (Naming the committed test file per §3 is not a path — it's the *location* of the evidence, so there is no collision.)
- **Incidental exactness where the requirement is approximate.** A condition demanding exactly 100 rows when the requirement is "about a page" fails compliant work on noise; where the requirement is approximate, `[auto]` conditions match tolerance or shape ("100 ± 5", "non-empty, sorted by date"), never a point value the spec doesn't actually require.
- **Wording two readers score differently.** The bar: two independent readers reach the same pass/fail verdict from the condition's text alone. If they could disagree, the condition will eventually produce both a false bounce and a false pin — rewrite until the text forces the verdict.

## 2. Done-conditions span three dimensions, not just invariants

The failure mode that ships thin features is a milestone whose done-conditions only assert *logic/invariants* (objectively checkable, so they get written) and leave *UX completeness* and *fidelity* implicit (so the agent fills them with generic defaults). Every **UI-touching** milestone's done-conditions must cover all three (a milestone with **no UI** — per the profile's Q8 "has UI?" — covers only the first two: **fidelity does not apply** and the design track is skipped):

- **Logic / invariants** — the data, the rules, the security/tenancy/gating guarantees. (What we always wrote.)
- **UX completeness** — the states and interactions that make the feature *feel* finished, enumerated, not assumed: the empty state, the loading state, the error state, the exact data each surface shows, the specific interactions (sort, filter, inline-edit, keyboard), the assistive-tech semantics (accessible names on icon-only controls, focus management in overlays, announcements for content that arrives on its own), responsive behavior at target widths, and the first-run (no-data) path. "A contact timeline" is not a done-condition; "the timeline groups events by day, shows dwell inline, filters by event type, and renders a themed empty state when the contact has no events" is.
- **Fidelity** — uses the design system's **tokens** (per the profile's token source + install verb, Q8.2 — no raw values bypassing the semantic tier), the **`design.md` fonts** (no system fallback), **themed components, not platform defaults**, the **`design.md`-recorded icon / chart / motion libraries** (no stray stock defaults), **and the layout matches this feature's own workbench composition** (see §6 — the real themed gallery per Q8.3, not a generic archetype, and not a static mockup). **Where the design system is type-led/low-color, two type-hierarchy floors are mandatory fidelity conditions** (type is the only hierarchy carrier there): a region rendering 3+ distinct text roles uses **≥2 distinct type levels** (size or weight), and **primary vs secondary actions are visually distinct**. For other systems the floors are not restated as a second rule — the sixth plan-pass question (§5) is the guard.

The split between mechanical and aesthetic still holds (§"Verification splits three ways" below): *whether the system was applied at all* — **plus the enumerated expression floors and signals** (the type-hierarchy floors above; §3's flatness signals) — is mechanical and verifier/walk-checkable; *does the composition sing* is still the human's at the review-feature gate.

**Tag every done-condition by who can check it** — `[auto]` (**a committed automated check asserts it** — schema, security/invariant, logic, git-tracking, and tests in *every* headless tier the profile names in Q11: unit **and component-render interaction tests**, not pure logic only), `[runtime]` (needs the **runtime walk** in §3: the surface activates in the *real* runtime on both builds, the live call actually works, the system was visually applied), or `[attended]` (human taste at `review-feature`). **A deterministic interaction — an open/close, a focus order, an edit that commits, a filter that re-derives — belongs in a committed tier, tagged `[auto]`; it is not `[runtime]` merely because it involves a UI.** The condition's deterministic core gets a committed test; only its divergence/live/fidelity remainder stays `[runtime]`. The tag is not decoration — it tells every build session, the `/verify-milestone` session, and the autonomous `verify-all-milestones` sweep **exactly which conditions it is structurally able to close, and which it must not write a pin without.** An untagged runtime/UX condition is precisely the gap that lets the offline suite pass green while the running app is broken (an earlier run's m4–m7: unit + invariant + build all green, three routes/actions dead at runtime). The autonomous parallel sweep can close `[auto]`; it cannot close `[runtime]`/`[attended]` (no browser, shared local services) — so a milestone carrying either must be pinned by a path that actually ran them, never by the sweep — except that under the profile's **proven Q13 isolation contract** the sweep closes `[runtime]` per-instance, scoped (activation + exploratory; fidelity only under a headless Q8.4 driver; live never); `[attended]` stays human, unconditionally.

## 3. Every milestone names its verification method and ground truth

In a form `verify-milestone` can read and dispatch without re-deciding. Examples:
- `verification: dynamic workflow over rent_roll.xlsx, criteria §4.2`
- `verification: verifier subagent against §3.1`

Method selection:
- **Dynamic workflow** when the check fans out over many independent units with a concrete ground-truth source (per-record/per-file/per-unit against a known-good dataset).
- **Verifier subagent** (`${CLAUDE_PLUGIN_ROOT}/agents/verifier.md`) for single-thread checks.
- **Committed tests across the profile's tiers (Q11) carry the enumerated assertions.** Every `[runtime]` condition **names the committed test** (in the highest tier the profile supports — component-render or end-to-end) that covers its *deterministic core*; the builder authors those tests as part of the build, and `verify-milestone` **runs** the suites rather than hand-driving each enumerated action. **This coverage is a hard pin gate:** a `[runtime]` condition whose named committed test is missing or failing forces verdict `blocked` and no pin — unless the profile records that tier as "n/a + why" (then the walk carries that condition alone, as the recorded exception). The point is artifact-based verification: a pin certifies *regression coverage*, not a point-in-time pass by an agent that leaves nothing behind (see profile-interface ⚠ Q11 — the agent walk as regression harness is the scar). **The split, in one line: the ladder governs the inner loop (build and debug route to the cheapest rung that can see the failure); the walk governs the gate (a pin is earned at full fidelity) — neither substitutes for the other.**
- **Runtime walk** (`[runtime]` conditions) appended for any milestone that adds/changes a **surface** (a route/screen/endpoint/command — per the project's stack profile, Q1–Q2) or a state-changing action — **narrowed to what only a live runtime proves**, on top of (never instead of) the committed tiers above. Driven by the project's **stack profile** (`specs/stack-profile.md`, derived against `${CLAUDE_PLUGIN_ROOT}/references/profile-interface.md`): seed a **no-human authenticated/test state** (Q5), then — **(a)** activate each new/changed surface and assert it's healthy (Q3: rendered with no error boundary / 2xx / clean exit, and no console/network errors) against **both the dev/fast build and the production build** (Q6 — they catch different breakage; await readiness and pass any required deploy-protection bypass); **(b)** for an AI/live milestone the **capped-key live variant** within the pre-authorized spend envelope (Q7), asserting invariants over nondeterministic output (never exact text; stays out of CI); **(c)** the **vision/fidelity diff** (the mechanical fidelity bullet below); and **(d)** a short **exploratory pass** — the unscripted layer that finds what no enumerated condition anticipated (an agent's comparative advantage; the enumerated actions are already covered by the committed suites it just ran). This **subsumes the old dev-mode + preview-deploy smokes**, and is the executable form of "render-proof before pin." It needs the profile's activation driver + seeded state and the shared local services **serially**, so it runs from the workflow/skill path (`/verify-milestone` or a feature e2e suite), **not** the plain `verifier` subagent, and **not** the parallel `verify-all-milestones` sweep — unless the profile's Q13 isolation contract is proven, in which case the sweep runs the scoped walk per-instance (the §2 scoping). *Why it exists:* a green unit suite cannot see the divergences this catches — the **test runtime differs from the production runtime** (a component/module-boundary error that renders standalone-and-passing but throws on the real surface) and the **dev build differs from the production build** (a transform-elided error) — both reach green pins and are invisible until the real thing runs (see profile-interface ⚠ Q3/Q6). Optionally, the walk may also produce richer evidence forms — a scripted browser video of the surfaces (agent-storyboarded, e.g. a shot-scraper-style recording, or whatever the stack's tooling supports) and a link to the verification session's transcript — which may accompany, never replace, the pin's evidence.
- **The poisoned-environment guard (a `[runtime]`/environment-dependent *negative* finding is gated on proof the environment reflects committed code).** Before a verifier or the runtime walk records a `[runtime]`/environment-dependent **negative** finding, it must first (a) **assert the working tree is clean for the profile's declared service-config paths** (Q14) — a dirty tree there means the running services reflect something other than committed code, so it is classified **environment, not a discrepancy** — and (b) **establish the profile's positive control** that the environment reflects committed state: probe the gate (restart the service, hit the endpoint, expect a real response), never trust a CLI start banner. **An independent signal change certifies a genuine reload; a repetition does not.** So reproducing a failure N times inside one un-restarted environment certifies nothing — N poisoned reads are one poisoned read — and the negative finding may stand only once an **independent signal** has gone from absent/stale to present. The specifics (which paths are service-config, which probe/signal proves a genuine reload) are the profile's (Q14); the requirement is the spine's. **Scope: `[runtime]`/environment-dependent *negative* findings only** — never an `[auto]`/static finding, and never a clean pass; the positive control is the price of a *negative* runtime verdict, not of every verdict.
- **Fidelity / vision check** (`[runtime]` for the mechanical part) — the runtime walk also screenshots each surface; an agent diffs the shots against the feature's **real workbench composition** (the fidelity reference per §6 — the workbench gallery composed into this feature's screens, Q8.3; **not** a static mockup, which is only an optional divergence sketch) and emits a deviation list — the application failures (system font where a display face is specified, unstyled native controls, tokens absent, wrong icon set, layout off) **plus three expression-level flatness signals: uniform type across a region with mixed semantic roles; adjacent equal-weight cards with no focal point; an action row of undifferentiated ghost buttons**. This is the *mechanical* fidelity check (was the system applied, plus the enumerated expression floors and signals — flat-but-token-correct no longer sails through) and is verifier/walk-side; the *aesthetic* call (does it sing) is `[attended]` at `review-feature`, which sits above it and only adjudicates the flagged deltas.
- Stakes scale rigor: financial logic and anything feeding a close gets the heavier mechanism when in doubt.
- **Hard-invariant milestones carry adversarial review as part of the gate, pre-pin — not a post-PR afterthought.** A milestone touching a hard invariant (auth/tenancy, gating, payments, data-migration) names `/security-review` as a **pre-pin** step in its `verification:` line; confirmed findings become regression tests + done-conditions *before* the `verified:` record is written. For an invariant milestone the verifier *attacks* the done-conditions (an earlier tenancy milestone proved cross-org SELECT was denied, yet a security review found a HIGH email-trust tenant join and a privilege-escalation the happy-path never probed). Pick the review type by kind: invariant/auth/data → `/security-review` (the vuln class); algorithmic/logic-heavy → `/code-review ultra` (the correctness class). The first-party **security-guidance** plugin is a useful *companion* while building a hard-invariant milestone in `implement-milestone` — it steers the build away from known vuln classes as the code is written — but it does not change this gate: `/security-review` remains the **pre-pin** step, run unchanged before the `verified:` record.

## 4. Size milestones, and make parallelizable real

Place boundaries where human review is cheap and being wrong is recoverable. Finer where correctness is critical (payments, auth, data migrations); coarser elsewhere. Mark independent milestones **parallelizable**.

**Size to the model's reliable horizon, not to a fixed rule.** A more capable model holds a longer chain of intent before it drifts, so where the stakes are low and being wrong is cheap to catch, a milestone can be *coarser* than habit suggests — fewer, larger units mean fewer merge/verify cycles for the same ground covered. This is a lens, not a license: it widens only along the "recoverable / cheap review" axis above. Where correctness is critical the boundaries stay fine regardless of model horizon (a payments milestone is small because the blast radius is large, not because the model can't hold more), and it never overrides the collision rule below — a coarser milestone that now touches a shared file is *more* likely to tangle, not less.

**Parallelizable is real only if the branches don't collide on a shared file.** Two milestones that both append to one growing file — a coverage registry, a shared test-helper module — are not independent at merge: every branch after the first conflicts, and a line-merge can *tangle* adjacent entries (an earlier run's fixture registry tangled three milestones' fixtures, forcing a hand-reconstruct + rebase + re-pin each — and with the SHA-pin gate, every rebase re-breaks the pin). Design any collector **file-per-entry with an auto-collecting barrel** (one `fixtures/<name>.ts` per entry, assembled by a glob/index) so registering is a *new file*, never an edit to a shared one. **Same fix for prose ledgers** (`decisions`, `deferrals`): file-per-entry directories, so parallel branches never collide. **Name the integration seams between sibling milestones up front** — when two share a runtime surface (a settings layout + a page that mounts in it, a nav + the route it links), that seam is invisible to each PR's own verification; plan a small integration milestone or require the post-wave reconciliation gate (an earlier run's M4 settings shell and M7 billing page double-wrapped the app chrome because neither saw the other).

**Every done-condition must be satisfiable within the run's pre-authorized envelope.** A condition needing mid-run human authorization (live API spend, a paid action) is, for an autonomous run, *unsatisfiable* — the same class of blocker as a missing credential; drain it the same way (pre-authorized before launch via provision, not gated per-use). If a gate genuinely can't be pre-authorized, it is a **planned stop point** — flag it on the milestone so the run halts there. **Never let an acceptance gate be silently deferred while later milestones build on top** — that stacks work on unverified code. Deferring a gate is allowed only as a conscious, `decisions/`-recorded choice.

## 5. The verified record — commit-pinned, one line per milestone

In the milestone's spec file:

`verified: <verdict> at <short-sha>, <date>, via <method> (evidence in PR #<n>)`

**The record is a hard pre-PR gate, not an intention.** Writing this line is the mandatory final step of `/verify-milestone` on a passing verdict, and a milestone's **code** PR is not opened unless its spec carries a `verified:` line whose pinned SHA is the verified code state. It is **CI-enforced** by a `verified-pin` job (`scripts/check-verified-pin.sh`, wired once by `spec-foundation`) that fails any milestone **code** PR whose `specs/milestones/<slug>.md` lacks a `verified:` line, or whose pinned `<short-sha>` is not an ancestor of HEAD, or where any non-plan file changed between that SHA and HEAD. "Verified code == merged code" is stronger and less brittle than a literal `HEAD^ == SHA`: a benign **plan-only** diff (a spec edit, a ledger keep-both, a carry-forward re-pin note) doesn't trip it, but real post-pin code drift does. **Plan paths = `specs/**` + `design/**`** (feature specs, milestone specs, mockups); everything else is code. The SHA pin makes the record self-invalidating — a record whose code moved past it describes an older state and needs re-verification. Status lives in the spec; evidence in the PR body. Nuanced verdicts stay nuanced ("clean except environment-skew items, see report") — never a bare checkmark. keel's shipped PreToolUse merge guard consults this same gate locally (deny on failure, ask on pass) as a **local backstop**; the CI `verified-pin` job remains the server-side enforcement and the hook never replaces it. Under an active autonomy mode, per-merge approval is delegated to the server-side required checks per `decisions/2026-07-autonomy-modes.md` — the pin gate itself is unchanged.

**After a pin merges, the milestone spec is historical and the merged code is authoritative.** The spec described the plan; once its `verified:` pin is on `main`, the code is the record of what was built. So when a later session finds spec and merged code disagreeing, **never edit correct merged code to match a stale spec** — the stale doc is what's wrong. Living docs (`specs/features/<feature>.md`, `00-product.md`, `01-architecture.md`) are reconciled to reality by the `land-feature` reconciliation step when the wave lands, and thereafter by the later milestone whose change invalidates them (that milestone owns the doc update, on its own branch — §4's file-per-entry discipline keeps it collision-free). The general "resolve in favor of the spec" rule governs *unbuilt* plan; it does not license rewriting shipped, verified code to chase a doc that history has already overtaken.

**No pin while a `[runtime]` (or required live) done-condition is unrun — and no pin without its committed coverage.** The pin certifies "verified code == merged code" *and* "every checkable condition actually ran green" *and* "the deterministic core is covered by committed tests that will keep running" — the last two are what the SHA gate alone can't see. A verdict is `clean`, and a `verified:` line may be written, only when every `[auto]` **and** every `[runtime]` condition has run green (the committed suites across the profile's Q11 tiers passed; the runtime walk passed on both dev and the production build; any required live call passed) **and** every `[runtime]` condition's named committed test exists and passes (§3 — profile-recorded "n/a + why" is the only exception). An unrun `[runtime]`/live condition forces verdict `blocked` and writes **no** pin. **A `verified:` line never carries a "pending verification" caveat** — if a check is pending, there is no pin (an earlier run's m4–m7's "Pending the fresh /verify-milestone session" pins were exactly this anti-pattern: pins written *before* the gate that catches the bugs ran; CI now rejects a `verified:` line containing "pending"/"unverified"/"to be verified" anywhere on the line **outside a backticked span** — the scan is the whole line minus backticked spans, so a domain term that merely contains a caveat word is written backticked, e.g. `` via e2e over the `pending-leads` suite ``, and passes). The single `[attended]` thing that lands *after* the pin is aesthetic approval at `review-feature` — it gates feature-done, not the milestone pin.

Beyond CI, a project *may optionally* wire this same gate as a **Stop hook** (`BASE_REF=origin/main scripts/check-verified-pin.sh`) so a code-PR session can't quietly end with un-pinned drift — a local backstop, never mandated and never a replacement for the server-side `verified-pin` job (`provision` step 4 records the wiring traps: a blocking Stop hook needs `exit 2` / a `{"decision":"block"}` wrapper, scope it to code-PR sessions, and keep its message clear of the neutrality denylist).

**The feature plan lands before its pins (the plan PR).** A `spec-feature` session's output — the feature spec, the mockups, and the milestone *specs* (done-conditions + the `verification:` line, **no** `verified:` pin yet) — ships as a **plan PR**: a plan-only change set (only `specs/**` + `design/**`, no code) that the gate **exempts** from the pin requirement. Each milestone's `verified:` pin is appended later, in that milestone's own **code** PR. Because the plan PR puts every milestone spec on `main` first, each later code PR diffs to touch only its *own* milestone file, so the per-file pin requirement still gates all code — no milestone's code reaches `main` unpinned (a PR that carries code is never plan-only, so it gets no exemption). This replaces the old "specs ride the first milestone branch" handling: the upfront plan is its own reviewable unit and is canonical on `main` before any build session reads it (which matters most under batched / by-wave cadence). *Run-discovered* spec/`decisions/`/`deferrals/` changes — those a milestone's build surfaces — still ride that milestone's branch, since they're part of that milestone's diff.

**A plan PR is gated on being well-formed, not on a pin (the adversarial plan pass).** The plan-only exemption above means a malformed plan would otherwise be caught a full build+verify cycle later, when `verify-milestone` bounces it — so the plan gets its own two-part gate before the PR opens. The **structural half is mechanized**: `scripts/check-plan.sh` (copied into the project by `spec-foundation` alongside the pin gate — copy, don't re-author) lints every milestone spec for a verification section, exactly one `[auto]`/`[runtime]`/`[attended]` tag per done-condition, a named committed test on every `[runtime]` condition, and a named reference wherever conditions mention fidelity — plus pin-line shape on chore batch specs; it runs in CI and passes on empty spec dirs, so it can be wired before the first spec exists. The **judgment half is not mechanizable**: before any plan PR opens, dispatch the read-only verifier against the drafted specs with the six plan-shaped questions — (1) does every surface/route have an owning milestone (§6); (2) does every interview decision trace to a done-condition; (3) are all applicable dimensions (§2) present per UI milestone; (4) are the integration seams named (§4); (5) flag any condition the verifier could not close *as written* — the two-readers bar (§1); (6) do each UI screen's fidelity conditions operationalize the design system's stated identity and hierarchy tools (per `specs/design.md`). Findings are fixed **in-session, before the PR opens** — never deferred to the build. This rule is owned here; `spec-feature` and `spec-change` carry one dispatch line each and do not restate it. None of this weakens the plan-only exemption: plan PRs still need no pin — they now need well-formed plans.

## 6. Layout is specified by reference, not prose — and the reference is the feature's *workbench composition*

A UI milestone points at its **workbench composition** as the canonical structure and carries "layout matches that composition" as the done-condition — it does **not** re-describe element placement in prose (which duplicates the reference and drifts).

The reference it points at is **the feature's own workbench composition** — the real, themed primitives (the workbench gallery, Q8.3) composed into this feature's screens by `spec-feature` when that feature was designed — **not a static mockup, and not a generic app-wide archetype.** **At spec time this reference is *named*, not built:** for a pure-composition feature it is the **named workbench/gallery components** the screens compose (plus, for a novel archetype only, an optional throwaway divergence sketch under `design/mockups/`); the composition is realized as real code at **build** time. If a real composed feature story/preview is genuinely wanted earlier, it is **design-gate infrastructure committed on `main`** — never code inside a plan-only PR (which is `specs/**` + `design/**` by definition, §5). This is the **diverge-cheap / converge-real** rule: a throwaway HTML mockup only picks a direction cheaply and is downgraded to an **optional divergence sketch**, never re-implemented as a spec; the moment a direction is chosen the feature converges into real themed components, and that *real composition* — not the mockup — is the fidelity reference a build matches and a verifier diffs against. This is the fix for "the build looks generic": a per-feature composition carries that feature's real intent; a shared archetype carries only structural consistency. A screen that is a novel archetype with no composed reference must get one *before* its milestone is authored, or it builds design-blind.

**Every route/screen in a feature is owned by exactly one milestone's done-conditions.** Enumerate the feature's routes and assign each — including the "obvious" detail/utility ones. A route no milestone owns is built by no one and checked by no one (an earlier run's `/location` route shipped as a stub because no done-condition claimed it). Carry the route→milestone map in the feature spec so coverage is auditable.

## 7. Verification splits three ways (the done-condition tags)

- **`[auto]` — a committed automated check asserts it.** Schema, security/invariant, logic, git-tracking, and the profile's Q11 tiers that run headlessly: unit **and component-render interaction tests** (deterministic interactions live here — see §2). Runnable by the `verifier` subagent and the parallel `verify-all-milestones` sweep. Pinnable by the sweep (for a milestone that also carries `[runtime]` conditions, only via the proven-Q13 path below).
- **`[runtime]` — the runtime walk (§3), on top of committed coverage.** The surface actually activates in the real runtime (dev + production build), the design system is actually applied (the vision diff), the live call actually works, plus the exploratory pass. Each `[runtime]` condition also **names the committed test** covering its deterministic core (§3) — missing coverage blocks the pin. Needs a browser + seeded DB → runs from `/verify-milestone` or a feature e2e suite, **not** the parallel sweep — unless the profile's Q13 isolation contract is proven, in which case the sweep closes it per-instance, scoped (activation + exploratory; fidelity only under a headless Q8.4 driver; live never). **Gates the pin** for any UI/action milestone.
- **`[attended]` — the human's.** Aesthetic approval at the design redline (per feature, in `spec-feature`) and at the post-build `review-feature` gate, now narrowed to adjudicating the vision diff's flagged deltas. "Did the system get applied + does it run + is it complete against spec" is `[auto]`+`[runtime]`; "does the composition sing" is `[attended]`. Don't put objective application/completeness/runtime failures on the human's side — that's what ships vanilla *and* what ships broken.

## 8. Test-authoring doctrine (cross-stack)

How the committed tests themselves are written — the rules the tiers above assume. Each was paid for at runtime and re-derived from chat before landing here; a builder authoring the conditions' tests (`implement-milestone` step 5) and a kickoff deriving the ladder (Q11) follow them. They are stack-neutral: no framework or runner is required by any of them, and a failure that recurs with a recognizable shape belongs in the profile's Q12 failure-signature table, not restated here.

1. **Per-suite data ownership.** Every suite owns its fixtures and cleans up after each test (a per-test teardown hook — an `afterEach`-style cleanup in whatever the stack's runner provides); shared fixtures across suites are **fragile by construction** — the recorded root of **five false-green/red incidents** in an earlier run. A suite **serialized to paper over cross-test contamination** is the *symptom* of violating this — a smell to surface and diagnose, never a configuration to inherit — and it is distinct from a stack ladder that legitimately serializes or isolates a shared-datastore tier by design (`references/feedback-ladders.md`'s "serialize or isolate" prior stands; the smell is contamination-driven serialization, not the deliberate tier).
2. **Run-unique fixture identity.** When a suite doesn't reset its datastore, fixture IDs are **run-unique** (a UUID or timestamp), never sequence counters — a counter collides with the previous run's leftovers, and the collision reads as either a spurious red or a false green against stale state. And an integration suite is run **twice** before the builder claims green: first-run luck is real. This is the builder's inner-loop discipline; verification-side flake handling stays where it is owned — **`verify-milestone`'s run-to-convergence hard rule**, in that skill's hard rules, unchanged and not restated here.
3. **Seed via the privileged path, or assert the round-trip.** Test seeds go through the superuser/direct path the profile records, or the suite asserts the seeded state read back equals what was written. The scar: an app-client seed **silently dropped a column it lacked the grant to write** — the suite ran green against a partial seed and the gap surfaced downstream.
4. **Cross-surface assertion parity.** A design-carrying behavior is asserted with **equal strength on every surface class the profile declares** (Q2); a presence-only assertion on the second surface is hollow coverage. The scar: a behavior fully asserted on one surface class shipped hollow on the other, because the second surface's test only checked that something rendered.
5. **One capped-key live test per AI/external-model module.** Env-gated — skipped when the key is absent — within the pre-authorized spend envelope (Q7), asserting **invariants, never text**. It is the rung that catches what every mock hides: a fenced-JSON parse failure lived exactly there, invisible below the live rung because the mocks returned clean JSON.
6. **Tests and walks name their server mode.** Some failure classes differ between the dev server and the production build — the recorded scar: an auth redirect that was a **401 on one and a 307 on the other** — so a test class known to straddle that line runs in **both columns**, and **the production-build column builds under the stack's environment** (the build-time env, not the dev shell). The runtime walk already covers both builds (Q6); this rule extends the *committed* tier where the class demands it.
7. **An undeclared, hand-run rung ships bugs.** A tier the ladder defers (e.g. mobile e2e without CI hardware) is recorded as a deferral **with an explicit risk note** — the `deferrals/` ledger is its home (cite it; its format lives there) — and load-bearing steps in that rung's scripts are never marked optional/skippable: **de-fang "optional: true" on anything that carries an assertion**. The scar: a keyboard-occlusion bug lived exactly in the hand-run mobile rung.
8. **Retry budgets are CI-only, waits are explicit.** Retries never mask flakes in the inner loop — retry count **0 locally, small in CI** — so a flake fails in front of the builder who can fix it, not in a green-after-retry CI log. And waits target **explicit conditions** (an element appearing, a response arriving), never network-idle heuristics — idle is a race dressed up as a wait, and it is the retry budget's accomplice.

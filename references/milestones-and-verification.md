# Milestone authoring + verification rules (shared)

The single source of truth for **how a milestone is written and how its "done" is checked.** Two skills point here so the rules never drift: `spec-foundation` *establishes* them once at kickoff (and wires the CI gate); `spec-feature` *applies* them every time it authors a feature's milestones. Neither skill restates these rules — they reference this file.

A **milestone** is the build/verify unit: one branch, one PR, one `verified:` pin, sized for one `implement-milestone` run. A **feature** (the unit `spec-feature` plans) fans out into 1–5 milestones. Everything below is per-milestone.

## 1. Every milestone's "done" is measurable

Write each completion condition so that a verifier subagent (`${CLAUDE_PLUGIN_ROOT}/agents/verifier.md`) can check it against the actual code/outputs, **and** it can be used verbatim as a `implement-milestone` completion condition.

"Auth works" fails this test. "POST /api/login with valid credentials returns 200 and a session cookie; invalid credentials return 401; covered by passing tests in auth.test.ts" passes it.

## 2. Done-conditions span three dimensions, not just invariants

The failure mode that ships thin features is a milestone whose done-conditions only assert *logic/invariants* (objectively checkable, so they get written) and leave *UX completeness* and *fidelity* implicit (so the agent fills them with generic defaults). Every **UI-touching** milestone's done-conditions must cover all three (a milestone with **no UI** — per the profile's Q8 "has UI?" — covers only the first two: **fidelity does not apply** and the design track is skipped):

- **Logic / invariants** — the data, the rules, the security/tenancy/gating guarantees. (What we always wrote.)
- **UX completeness** — the states and interactions that make the feature *feel* finished, enumerated, not assumed: the empty state, the loading state, the error state, the exact data each surface shows, the specific interactions (sort, filter, inline-edit, keyboard), responsive behavior at target widths, and the first-run (no-data) path. "A contact timeline" is not a done-condition; "the timeline groups events by day, shows dwell inline, filters by event type, and renders a themed empty state when the contact has no events" is.
- **Fidelity** — uses `@theme` tokens (no raw hex), the `design.md` fonts (no system fallback), themed components (no native `<select>`), the `design.md` icon set + chart library (no stray Lucide/stock Recharts), **and the layout matches this feature's own mockup** (see §6 — points at the feature mockup, not a generic archetype).

The split between mechanical and aesthetic still holds (§"verification splits in two" below): *whether the system was applied at all* is mechanical and verifier/walk-checkable; *does the composition sing* is the human's at the review-feature gate.

**Tag every done-condition by who can check it** — `[auto]` (the verifier / static checks: schema, RLS, unit tests, logic, git-tracking), `[runtime]` (needs the **runtime walk** in §3: a route actually renders, a server action actually runs, the live call actually works), or `[attended]` (human taste at `review-feature`). The tag is not decoration — it tells every build session, the `/verify-milestone` session, and the autonomous `verify-all-milestones` sweep **exactly which conditions it is structurally able to close, and which it must not write a pin without.** An untagged runtime/UX condition is precisely the gap that lets the offline suite pass green while the running app is broken (Crelaunch m4–m7: unit + RLS + build all green, three routes/actions dead at runtime). The autonomous parallel sweep can close `[auto]`; it cannot close `[runtime]`/`[attended]` (no browser, shared local services) — so a milestone carrying either must be pinned by a path that actually ran them, never by the sweep.

## 3. Every milestone names its verification method and ground truth

In a form `verify-milestone` can read and dispatch without re-deciding. Examples:
- `verification: dynamic workflow over rent_roll.xlsx, criteria §4.2`
- `verification: verifier subagent against §3.1`

Method selection:
- **Dynamic workflow** when the check fans out over many independent units with a concrete ground-truth source (per-record/per-file/per-unit against a known-good dataset).
- **Verifier subagent** (`${CLAUDE_PLUGIN_ROOT}/agents/verifier.md`) for single-thread checks.
- **Runtime walk** (`[runtime]` conditions) appended for any milestone that adds/changes a **surface** (a route/screen/endpoint/command — per the project's stack profile, Q1–Q2) or a state-changing action — the autonomous proof the thing actually *runs*, not just that its units pass. Driven by the project's **stack profile** (`specs/stack-profile.md`, derived against `${CLAUDE_PLUGIN_ROOT}/references/profile-interface.md`): seed a **no-human authenticated/test state** (Q5), then for each new/changed surface — **(a)** activate it and assert it's healthy (Q3: rendered with no error boundary / 2xx / clean exit, and no console/network errors), and **(b)** drive each new action and assert its **effect** (Q4). Run against **both the dev/fast build and the production build** (Q6 — they catch different breakage; await readiness and pass any required deploy-protection bypass), and for an AI/live milestone the **capped-key live variant** within the pre-authorized spend envelope (Q7), asserting invariants over nondeterministic output (never exact text); the live variant stays out of CI. This **subsumes the old dev-mode + preview-deploy smokes and the UX walk's correctness role**, and is the executable form of "render-proof before pin." It needs the profile's activation driver + seeded state and the shared local services **serially**, so it runs from the workflow/skill path (`/verify-milestone` or a feature e2e suite), **not** the plain `verifier` subagent, and **not** the parallel `verify-all-milestones` sweep. *Why it exists:* a green unit suite cannot see the divergences this catches — the **test runtime differs from the production runtime** (a component/module-boundary error that renders standalone-and-passing but throws on the real surface) and the **dev build differs from the production build** (a transform-elided error) — both reach green pins and are invisible until the real thing runs (see profile-interface ⚠ Q3/Q6).
- **Fidelity / vision check** (`[runtime]` for the mechanical part) — the runtime walk also screenshots each surface; an agent diffs the shots against the feature's mockups (`design/mockups/<feature>/**`) and emits a deviation list (system font where a display face is specified, unstyled native controls, tokens absent, wrong icon set, layout off). This is the *mechanical* fidelity check (was the system applied) and is verifier/walk-side; the *aesthetic* call (does it sing) is `[attended]` at `review-feature`, which sits above it and only adjudicates the flagged deltas.
- Stakes scale rigor: financial logic and anything feeding a close gets the heavier mechanism when in doubt.
- **Hard-invariant milestones carry adversarial review as part of the gate, pre-pin — not a post-PR afterthought.** A milestone touching a hard invariant (auth/tenancy/RLS, gating, payments, data-migration) names `/security-review` as a **pre-pin** step in its `verification:` line; confirmed findings become regression tests + done-conditions *before* the `verified:` record is written. For an invariant milestone the verifier *attacks* the done-conditions (a Crelaunch RLS milestone proved cross-org SELECT was denied, yet a security review found a HIGH email-trust tenant join and a privilege-escalation the happy-path never probed). Pick the review type by kind: invariant/auth/data → `/security-review` (the vuln class); algorithmic/logic-heavy → `/code-review ultra` (the correctness class).

## 4. Size milestones, and make parallelizable real

Place boundaries where human review is cheap and being wrong is recoverable. Finer where correctness is critical (payments, auth, data migrations); coarser elsewhere. Mark independent milestones **parallelizable**.

**Parallelizable is real only if the branches don't collide on a shared file.** Two milestones that both append to one growing file — a coverage registry, a shared test-helper module — are not independent at merge: every branch after the first conflicts, and a line-merge can *tangle* adjacent entries (the cre-launch Wave-2 RLS registry tangled three milestones' fixtures, forcing a hand-reconstruct + rebase + re-pin each — and with the SHA-pin gate, every rebase re-breaks the pin). Design any collector **file-per-entry with an auto-collecting barrel** (one `fixtures/<name>.ts` per entry, assembled by a glob/index) so registering is a *new file*, never an edit to a shared one. **Same fix for prose ledgers** (`decisions`, `deferrals`): file-per-entry directories, so parallel branches never collide. **Name the integration seams between sibling milestones up front** — when two share a runtime surface (a settings layout + a page that mounts in it, a nav + the route it links), that seam is invisible to each PR's own verification; plan a small integration milestone or require the post-wave reconciliation gate (cre-launch's M4 settings shell and M7 billing page double-wrapped the app chrome because neither saw the other). See [[shared-append-harness-file-wave-conflict]].

**Every done-condition must be satisfiable within the run's pre-authorized envelope.** A condition needing mid-run human authorization (live API spend, a paid action) is, for an autonomous run, *unsatisfiable* — the same class of blocker as a missing credential; drain it the same way (pre-authorized before launch via provision, not gated per-use). If a gate genuinely can't be pre-authorized, it is a **planned stop point** — flag it on the milestone so the run halts there. **Never let an acceptance gate be silently deferred while later milestones build on top** — that stacks work on unverified code (see [[fable5-acceptance-gates-stop-not-defer]]). Deferring a gate is allowed only as a conscious, `decisions/`-recorded choice.

## 5. The verified record — commit-pinned, one line per milestone

In the milestone's spec file:

`verified: <verdict> at <short-sha>, <date>, via <method> (evidence in PR #<n>)`

**The record is a hard pre-PR gate, not an intention.** Writing this line is the mandatory final step of `/verify-milestone` on a passing verdict, and a milestone's **code** PR is not opened unless its spec carries a `verified:` line whose pinned SHA is the verified code state. It is **CI-enforced** by a `verified-pin` job (`scripts/check-verified-pin.sh`, wired once by `spec-foundation`) that fails any milestone **code** PR whose `specs/milestones/<slug>.md` lacks a `verified:` line, or whose pinned `<short-sha>` is not an ancestor of HEAD, or where any non-plan file changed between that SHA and HEAD. "Verified code == merged code" is stronger and less brittle than a literal `HEAD^ == SHA`: a benign **plan-only** diff (a spec edit, a ledger keep-both, a carry-forward re-pin note) doesn't trip it, but real post-pin code drift does. **Plan paths = `specs/**` + `design/**`** (feature specs, milestone specs, mockups); everything else is code. The SHA pin makes the record self-invalidating — a record whose code moved past it describes an older state and needs re-verification. Status lives in the spec; evidence in the PR body. Nuanced verdicts stay nuanced ("clean except environment-skew items, see report") — never a bare checkmark.

**No pin while a `[runtime]` (or required live) done-condition is unrun.** The pin certifies "verified code == merged code" *and* "every checkable condition actually ran green" — the second half is what the SHA gate alone can't see. A verdict is `clean`, and a `verified:` line may be written, only when every `[auto]` **and** every `[runtime]` condition has run green (the runtime walk passed on both dev and the production build; any required live call passed). An unrun `[runtime]`/live condition forces verdict `blocked` and writes **no** pin. **A `verified:` line never carries a "pending verification" caveat** — if a check is pending, there is no pin (Crelaunch m4–m7's "Pending the fresh /verify-milestone session" pins were exactly this anti-pattern: pins written *before* the gate that catches the bugs ran; CI now rejects a `verified:` line whose verdict text contains "pending"/"unverified"/"to be verified"). The single `[attended]` thing that lands *after* the pin is aesthetic approval at `review-feature` — it gates feature-done, not the milestone pin.

**The feature plan lands before its pins (the plan PR).** A `spec-feature` session's output — the feature spec, the mockups, and the milestone *specs* (done-conditions + the `verification:` line, **no** `verified:` pin yet) — ships as a **plan PR**: a plan-only change set (only `specs/**` + `design/**`, no code) that the gate **exempts** from the pin requirement. Each milestone's `verified:` pin is appended later, in that milestone's own **code** PR. Because the plan PR puts every milestone spec on `main` first, each later code PR diffs to touch only its *own* milestone file, so the per-file pin requirement still gates all code — no milestone's code reaches `main` unpinned (a PR that carries code is never plan-only, so it gets no exemption). This replaces the old "specs ride the first milestone branch" handling: the upfront plan is its own reviewable unit and is canonical on `main` before any build session reads it (which matters most under batched / by-wave cadence). *Run-discovered* spec/`decisions/`/`deferrals/` changes — those a milestone's build surfaces — still ride that milestone's branch, since they're part of that milestone's diff.

## 6. Layout is specified by reference, not prose — and the reference is the *feature's own* mockup

A UI milestone points at its mockup as the canonical structure and carries "layout matches that structure" as the done-condition — it does **not** re-describe element placement in prose (which duplicates the mockup and drifts).

The mockup it points at is **the feature's own screen mockup** (`design/mockups/<feature>/<screen>.html`, produced by `spec-feature` when that feature was designed), not a generic app-wide archetype. This is the fix for "the build looks generic": a per-feature mockup carries that feature's real intent; a shared archetype carries only structural consistency. A screen that is a novel archetype with no mockup must get one added *before* its milestone is authored, or it builds design-blind.

**Every route/screen in a feature is owned by exactly one milestone's done-conditions.** Enumerate the feature's routes and assign each — including the "obvious" detail/utility ones. A route no milestone owns is built by no one and checked by no one (Meridian's `/location` shipped as a stub because no done-condition claimed it). Carry the route→milestone map in the feature spec so coverage is auditable.

## 7. Verification splits three ways (the done-condition tags)

- **`[auto]` — the verifier / static checks.** Schema, RLS, unit/integration tests, logic, git-tracking. Runnable by the `verifier` subagent and the parallel `verify-all-milestones` sweep. Pinnable by the sweep.
- **`[runtime]` — the runtime walk (§3).** A route actually renders, a server action actually runs, the enumerated states render, the design system is actually applied (the vision diff), the live call actually works. Needs a browser + seeded DB → runs from `/verify-milestone` or a feature e2e suite, **not** the parallel sweep. **Gates the pin** for any UI/action milestone.
- **`[attended]` — the human's.** Aesthetic approval at the design redline (per feature, in `spec-feature`) and at the post-build `review-feature` gate, now narrowed to adjudicating the vision diff's flagged deltas. "Did the system get applied + does it run + is it complete against spec" is `[auto]`+`[runtime]`; "does the composition sing" is `[attended]`. Don't put objective application/completeness/runtime failures on the human's side — that's what ships vanilla *and* what ships broken.

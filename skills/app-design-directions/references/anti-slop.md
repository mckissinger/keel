# Anti-Slop Reference: The AI Design Fingerprint

## Why this file exists

LLMs converge on the statistical median of their training data. The result is a recognizable "generated by a chat interface without an opinion" look. A 2026 audit of 1,590 Show HN landing pages found 22% triggered four or more of these patterns and only 46% were clean — and reviewers, customers, and investors have learned to spot the fingerprint. These patterns are banned **as defaults**: each is acceptable if deliberately chosen for a specific reason tied to this app's brief, and unacceptable if it appears because it's what would have been produced for any app.

The test for every choice: *would I have made this exact choice for a completely different app?* If yes, it's a default, not a decision. Go back to the discovery brief and derive something from this app's actual world.

## The fingerprint patterns

**These patterns are calibrated to current-generation models (as of 2026-07) and decay as
models retrain.** Named default-cluster blocklists shift convergence rather than eliminate it:
ban today's clusters and the next model generation converges on the next nameable one. Expect
this list to be refreshed periodically; the durable complement that survives retraining is the
cross-direction convergence self-check below, which needs no named patterns at all.

### Typography tells
1. **Inter for everything**, especially a centered hero/page title. Inter is a fine typeface that has become the Helvetica of the LLM era — its presence alone now signals "generated."
2. **The recycled pairings**: Space Grotesk, Instrument Serif, Geist, used the way every AI output uses them.
3. **Serif-italic accent on one word** in an otherwise-sans headline ("Ship *faster*").
4. **All-caps section labels and headings everywhere** (small, letterspaced, grey). One eyebrow label can be a tool; a page covered in them is the fingerprint.

### Color tells
5. **"VibeCode purple"** — the lavender-purple accent that leaks out of most generation. Purple/violet gradients on white doubly so.
6. **Permanent dark mode by default** — dark theme chosen for no reason, with medium-grey body text. (34% of audited pages. Dark mode is a legitimate *user setting* and legitimate for some tools — monitoring dashboards, media — but it is not a personality.)
7. **Barely-passing or failing contrast**, especially grey-on-dark body text. This one is also a functional defect — check WCAG AA (4.5:1 body, 3:1 large text) always.
8. **Gradients as decoration** — background washes, gradient text, gradient buttons with no semantic job.
9. **Large colored glows / colored box-shadows** around cards and buttons.

### Layout tells
10. **Centered hero in a generic sans + pill badge sitting right above the H1.**
11. **Colored left/top borders on cards** — described by designers as "almost as reliable a sign of AI-generated design as em-dashes for text."
12. **Identical icon-card grids** — three or four equal cards, icon on top, short blurb (22% of audited pages).
13. **Numbered 1-2-3 step sequences** used decoratively when the content isn't actually a sequence.
14. **Stat banner rows** (big number, small label, ×4).
15. **Emoji as icons** in navs, sidebars, lists, or code comments.
16. **8px-radius-on-everything faux-minimalism** — uniform rounding, uniform shadow recipe, no considered shape language.

### CSS/system tells
17. **Stock shadcn/ui defaults shipped unmodified** — the zinc palette, default radius, default shadows. shadcn is an excellent foundation *because* you own the code; shipping it untouched is the fingerprint. Retheme the tokens.
18. **Glassmorphism** (blur + transparency + thin light border) as the default surface treatment.
19. **Inconsistent depth grammar** — every component carrying a different shadow recipe because each was generated independently.

### Material-palette tells (the library defaults)
These are the library choices that go unmade and silently default to the generic look. Each direction must choose *against* them.
23. **Lucide-everywhere** — the default shadcn icon set used unconsidered, uniform thin strokes on every action. Pick an icon set + weight that belongs to the direction (Phosphor, Heroicons, Tabler, a duotone/filled style) and use it deliberately.
24. **Stock Recharts/Chart.js** — default axes, default gridlines, default category palette, default tooltip. The most reliable "generic dashboard" tell. Charts must wear the direction's palette, type, and density (restyle Recharts/Tremor/visx, or pick a library that themes cleanly).
25. **Default-styled or absent motion** — every interaction either instant or carrying the framework's default ease/duration. Decide a motion stance (none / functional / expressive) and a library, and apply it consistently — including `prefers-reduced-motion`.

### The "tasteful AI" cluster (subtler, newer)
These are the looks models produce when explicitly told to avoid the list above. They are second-order defaults — still avoid as unconsidered choices:
20. **Warm cream background (~#F4F1EA) + high-contrast serif display + terracotta accent.**
21. **Near-black background + single acid-green or vermilion accent.**
22. **Broadsheet/editorial look**: hairline rules, zero radius, dense newspaper columns — applied to apps that aren't editorial.

## Product-UI-specific slop (the deeper failure)

Visual tells are surface. In apps, the deadlier slop is structural — "technically correct but devoid of specific identity":

- **Everything is a card.** Tables become card grids; dense comparison work becomes scroll-heavy air. Density slop is the #1 product-UI failure: AI defaults to whitespace-rich layouts even for professional daily-driver tools.
- **Missing state grammar.** Focus, disabled, error, empty, and loading states absent or improvised per-component instead of designed as a system. Static polish masking the lack of rules underneath.
- **Generic flows.** Interchangeable UX patterns pasted in without asking what this user's actual job on this screen is.
- **Copy that fits the frame but misses the job.** Labels named after how the system is built ("Webhook config") instead of what the person controls ("Notifications").
- **Non-systematic values.** Arbitrary margins and one-off spacing that don't come from a scale — invisible in a static screenshot, fatal as the product grows.

## Native / mobile slop (the platform-feel failure)

When the platform target is **mobile or cross-platform** (Phase 0, from the profile's Q8.6 platform-convention set), a second family of defaults appears — the tell that a web design was pasted onto a native surface. These are banned **as defaults** exactly like the web list above: acceptable only if a deliberate, brief-justified reason is written down, unacceptable when they appear because a web look was the path of least resistance. The brand ports across platforms; the native feel does not — so imposing web conventions on native is the mobile equivalent of shipping stock shadcn.

- **A web look forced onto native.** A desktop-style top nav bar, a hover-dependent affordance (native touch surfaces have no hover), dense pointer-scaled hit targets, or a fixed 1280–1440 canvas on a phone. If the layout only makes sense with a mouse, it is web slop on a native surface.
- **Custom-drawn controls where the platform has its own.** Re-implementing a picker, date-picker, switch, segmented control, or share sheet from scratch instead of theming the platform's native control (iOS HIG / Android Material). Users expect the system control's behavior, accessibility, and muscle memory; a look-alike is uncanny and usually less accessible.
- **Missing or wrong navigation model.** Ignoring the platform's navigation grammar — iOS back-swipe and navigation stack, Android's system back and Material navigation, the tab-bar vs. nav-drawer conventions — or inventing a bespoke one. Navigation is where "this doesn't feel native" is felt first.
- **Absent or incorrect gestures.** No swipe-to-go-back, no pull-to-refresh where the platform implies it, or gestures wired against the platform's direction/semantics. Gestures are part of the native contract, not decoration.
- **System affordances ignored.** Safe-area insets, the notch/dynamic-island, home indicator, keyboard avoidance, dark-mode and dynamic-type/text-scaling, and the accessibility reduce-motion flag not respected — the surface reads as a web page in a shell, not a native app.

The inversion still holds: brand-universal choices (the token palette, type scale, iconography, information architecture) **should** carry across platforms unchanged — porting those is correct, not slop. It is only the **native-feel** layer (navigation, gestures, system controls) that must be native per platform.

## The cross-direction convergence self-check (the durable complement)

The fingerprint list above names *known* defaults; this check catches the *unknown* ones — and
because it compares directions against each other rather than against a dated list, it survives
model retraining. After drafting the Phase 1 directions, compare them pairwise: **when two
drafted directions resemble each other** — same depth grammar, same density stance, same type
system wearing two palettes — **treat the resemblance as a shared default, not a coincidence**,
and discard or rework at least one of the pair until the set is genuinely divergent. Whatever
you produced twice unprompted is what the model produces for *any* app; convergence inside your
own draft set is the fingerprint diagnosing itself.

## How to use this file

In Phase 1, run every direction spec down this list. For each hit, either (a) replace it with a choice derived from the app's domain, or (b) write one sentence justifying it as a deliberate fit for *this* brief ("dark theme because this is a market-monitoring tool used on a second screen all day"). A justified pattern is fine. An unjustified one means the direction needs another pass.

And remember the inversion: the user's explicit taste always wins. If they ask for a dark UI with purple, that's a brief, not slop — execute it with more craft than the median would.

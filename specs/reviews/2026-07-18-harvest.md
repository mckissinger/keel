# Tier A — defects in the harvest verb itself

These determine whether any future harvest is trustworthy, so they rank first.

## A1. Sources are matched by directory string, so a moved checkout silently drops a project
**Status: new.** **Grain: `spec-change`** (`harvest` + cursor format).

The cursor listed 4 sources. **Ten were active this week.** Two of the three largest corpora had
never been mined at all:

- **CRE Launch** moved to `~/Dev Projects/cre-list/`, so its sessions land in an unlisted
  directory. The first sweep read the old path, found nothing since Jun 22, and reported the
  project *dormant* — while 13 uncovered sessions sat next door, containing exactly the E2E
  material the run was commissioned to find.
- **BidLevel** (`test-proj-1`) has **18 sessions this week**, is fully keel-managed and unusually
  mature in its use of keel, and has never appeared in the cursor in any form.

A moved checkout is indistinguishable from an inactive project under string matching. The
cursor's own seed note shows the pattern was known — it records one project spanning two dirs
"because the checkout moved" — and the handling was never generalized.

**Fix**: resolve by project identity. At minimum, every run must enumerate all dirs under
`~/.claude/projects/` with sessions newer than its floor and **reconcile against the cursor**,
reporting unlisted sources rather than mining only what's listed. "Source went quiet" must be an
assertion, never an assumption.

## A2. Not every `type=="user"` entry is a human
**Status: new.** **Grain: `spec-change`** (`harvest` signal ladder).
Flagged in CRE `d89b3abb`; independently visible in BidLevel `2f7dba4c` (a session with zero
human text).

**Verified directly, and larger than first reported.** User-role entries in that session carry an
`origin.kind` field: **26 `human`, 46 `task-notification`, and 545 with no origin field at all.**
A naive `type=="user"` extraction therefore ingests roughly 24× more than actual human input.
Scheduler fallbacks, `/loop` re-fires, task notifications, and local-command stdout all land in
the user role.

One miner caught this and filtered on `origin.kind == "human"`; the others weren't told to. Any
finding of the form "the user asked twice" is unsafe until that filter is standard — which means
several frequency claims in this digest (E4's instance counts especially) should be re-derived
once the filter exists.

# Tier B — verification integrity

The week's most expensive failures. All four are cases where **a gate reported green, or red, for
reasons unrelated to the code**.

## B1. Environment state masquerading as code defects — the dominant failure class
**Status: new.** **Grain: `spec-change`** (`verify-milestone` environment precondition).
BidLevel `fc2160d0`; Jarvis `4020ae55`; corroborated in Relay `df3b301b`.

Three independent instances in one week, in three different projects:

- **BidLevel**: verifying M1 of `auth-redesign` hit red e2e and `captcha_failed` on
  `POST /auth/v1/otp`. Cause: the working tree carried **uncommitted `supabase/config.toml`
  captcha blocks belonging to M3**, rejecting M1's correctly token-less sends. The transcript's
  own words: *"an environment failure that reads exactly like a code defect."* It explicitly
  predicts recurrence when M2 is verified against M3's block.
- **Jarvis**: a `.env.local` copied from another machine made vitest load a **hosted**
  service-role key against the **local** stack — every DB test failed "permission denied."
- **Relay**: a Release build crashed on a missing env var, diagnosed correctly as a procedure
  gap rather than a milestone defect.

**Fix shape** (BidLevel's own remedy, worth promoting): assert the working tree is clean for
service-config paths before verifying, and **probe the gate rather than trusting the CLI's start
banner** — restart the stack, then hit the endpoint and expect a real response.

## B2. A verifier was confidently, reproducibly wrong — reproduction count mistaken for confidence
**Status: new.** **Grain: reference update** (verifier contract).
BidLevel `fc2160d0`.

Downstream of B1: a verifier reported "no 6-digit code found," **reproduced 4× across dev+prod
and new+existing users**, and concluded users had no code to type. After the stack was restarted
on committed config, the same probe returned real codes immediately.

The tell that finally distinguished a genuine reload was an *email subject change* — a positive
control, not a repetition. **Reproducing a failure N times inside one poisoned environment is
worth nothing.** A verifier should be required to establish an independent positive control
proving the environment reflects committed state **before a negative finding is permitted to
stand.**

This is the sharpest new finding in the corpus. It inverts an assumption the whole verification
model rests on.

## B3. `reuseExistingServer` adopts a stale server — green runs that prove nothing
**Status: new.** **Grain: `spec-change`** (runtime-proof contract in `references/profile-interface.md`).
Relay `b4a08064`; Jarvis `4020ae55`; CRE `06d58e96`.

Three projects, three variants of one hole:

- Relay: `next start` never rebuilds, so a render milestone's e2e passed against a `.next`
  predating the change — the fidelity screenshot showed the **old design**. Behavior-only
  assertions pass identically on old and new render, so nothing flags it.
- Jarvis: Playwright adopted a leftover server on the port, so the "production" suite asserted
  against a different build entirely — **twice**.
- CRE: same class, mitigated by a remembered `rm -rf .next` before prod walks.

Jarvis landed the right fix with the reasoning committed: **`reuseExistingServer: false`** —
failing loudly on a busy port is the correct outcome. Keel's runtime-proof contract should
forbid server adoption outright rather than leaving each project to rediscover it.

## B4. The milestone-scoped filter is baked into the orchestration prompts
**Status: partially addressed.** **Grain: `spec-change`** (`implement-feature` + `verify-milestone` dispatch prompts).
Relay `b4a08064`/`4c253c14`/`df3b301b`/`5d6f958a`; CRE `663f3a8e`/`06d58e96`.

Reproduces independently in both mature projects. Relay: seven follow-ons, including a
three-branch re-pin cascade; `implement-feature` dispatched **both** builder and fresh-context
verifier with milestone-scoped e2e, and a builder was caught reporting *"I did not run e2e (per
instruction)."* CRE, textbook: a filtered RLS run showed **6 passed / 192 skipped** — green; the
full run showed **4 failed**; clean `main` showed **6 failed** — a pre-existing failure no scoped
run could see.

`test-health` exists to *audit* for this but doesn't touch the dispatch prompts that cause it.
The verifier dispatch must **forbid** a spec filter outright.

## B5. `lessons/` files don't escalate
**Status: new.** **Grain: reference update.**
Relay `5d6f958a`/`df3b301b`; CRE `54447188`.

`full-e2e-suite-when-touching-shared-render-path.md` reached **five recorded repeats** before
anything structural happened — and had already concluded, after repeat four, that this should be
"a hard pre-pin checklist item, not a judgment call." It then recurred three more times.
Documentation is not an enforcement mechanism.

**Superseded — do not build the repeat-count threshold.** An earlier draft proposed a generic
mechanism promoting a lesson into a remediation milestone after N repeats. Replace it with CRE's
concrete practice: flakes tracked as **files carrying measured reproduction rates** (`2/6 clean
main`, `1/5 prod only`, `~50 runs to slug exhaustion`), escalated by hit count, closed in a batch
chore. It gives a verifier something to classify against instead of re-litigating "is this real?"
each session, and it is proven in use rather than invented here.

**Correction on verification**: those flake files are **not** in CRE's `deferrals/` today — that
directory holds feature deferrals. They were closed by the batch chore
`specs/chores/e2e-harness-reliability.md`. The practice was therefore a **one-time batch, not a
standing ledger**, which makes this item *adoption of a good pattern* rather than remediation of
a recurring defect. Justify it on those terms.

# Tier C — the E2E cost question

## C1. Relay already ran the experiment CRE needs — same root cause, opposite outcomes
**Status: new.** **Grain: `test-health` in the CRE repo** (remediation is CRE-side).

**The direct answer to the question that commissioned this run.** Both suites had `workers: 1`
plus one shared seeded user, for the same stated reason. Only one fixed it.

| | Relay (before) | Relay (after) | CRE (today) |
| --- | --- | --- | --- |
| Parallelism | `workers: 1` | **6 workers** | `workers: 1`, `fullyParallel: false` |
| Isolation | `afterEach` + alphabetical order | per-test agent isolation | `freshOrg` + FK cascade, no reset |
| Full run | 13 specs serial | **13/13 in ~3s**, 3× zero-flake | 53 specs / ~206 tests, **~10.2m local** |

Verified today in CRE's live config — the comment naming the shared-fixture cause is still there.

Relay's fix was structural: delete the coupling, parallelism comes free. CRE sharded instead
(`decisions/2026-07-09-e2e-ci-sharding.md`), explicitly because it required no test changes.
Defensible then, but it bought time rather than fixing the cause, and the cost regrew: sharding
projected ~8 min; **shard 1 now runs 20–24 min and gates every PR**, with shards imbalanced
(21m / 16m / 12m) so 30–40% of the fleet idles. CRE's unit tier does 1,700 tests in 16 seconds —
a ~40× gap.

Keel's own doctrine already names this, and was loaded in a CRE session (`54447188`) without
binding: a suite *"serialized to paper over cross-test contamination is the symptom — a smell to
surface and diagnose, never a configuration to inherit."*

**Rebalance the shards first** (~30% off PR wall time, zero test changes), then diagnose
`workers: 1`. Relay is the existence proof.

## C2. `[runtime]` conditions have no cadence contract
**Status: new.** **Grain: `spec-change`** (`references/milestones-and-verification.md` + `verify-milestone`).
Relay `310d7574`; corroborated across all CRE sessions.

The pin gate requires every `[runtime]` condition to have run green *for the commit being
pinned*, so a scheduled suite can **never** close one. Keel *"underspecifies cadence, and the gap
got filled with the most expensive option."* CRE is that gap concrete: **no nightly or on-demand
lane exists** — every PR pays ~20–24m, including after every rebase and force-push.

Verified absent at HEAD: no `workflow_dispatch`, `nightly`, or cadence handling in
`references/milestones-and-verification.md`.

Proposed shape: a `[runtime]` condition names its trigger (`on-demand` / `per-PR` /
`nightly+dispatch`), and `verify-milestone` **summons** a run for the commit under pin and cites
it. Two caveats worth recording before building it: (a) a milestone is roughly one PR and exactly
one pin, so per-pin ≈ per-push in the base case — the saving is on rebases and review iterations,
not elimination; (b) a red nightly against `main` invalidates no pin and has **no defined
response** — decide what it does before adding it.

## C3. Long suites break the agent supervision model
**Status: new.** **Grain: `spec-change`** (suite-execution rule).
CRE `06d58e96`; BidLevel `01dc289a`/`b7a7bb46`; Relay (Monitor timeouts).

Subagents backgrounded a 10-minute suite and burned budget waiting: **336k tokens / 31.8 min**,
**320k / 40.3 min**, and **325k tokens across 5 tool uses** — pure waiting. Reported results
amounted to "still running." The user polled repeatedly (*"status?"*, *"?"*, *"please confirm it
is still running? I may have accidentally shut it off"*). The correction was to run **foreground
with a 900000 ms timeout**. `Monitor` timed out on long runs at least four times across projects.

A suite whose expected duration exceeds the monitor budget should run foreground with an explicit
timeout, not backgrounded. Cheap rule, large saving.

# Tier D — environment and provisioning

## D1. The plugin dogfood loop is broken — a merged skill is not an invocable skill
**Status: new — and this session hit it live.** **Grain: `spec-change`** (a release/reload contract).
keel worktrees `1f5bd333`, `ef403031`, `ce036f4a`, `c4c8f153`.

Skills authored in a worktree are invisible to the running session, because sessions resolve keel
skills from the immutable install cache. A skill can be **committed, merged, released, and
tagged** and `/logo` still won't route. `/reload-plugins` updates the on-disk cache but **does not
refresh a booted session's skill registry** — the session keeps its boot-time snapshot.

Three-way version skew was live simultaneously: worktree manifest at `1.15.0`, install path at
`1.18.0`, session serving skills from `1.14.0`/`1.17.0`. The in-repo `plugin.json` silently rots
because the installer versions from marketplace/tag metadata, not from the manifest.

Cost: the user hit it in three separate sessions, with visible frustration by the third
(*"We've done this twice"*), and roughly **three abandoned session starts** trace to it.

The irony worth recording: it *forces* violating the standing lesson
`keel-invoke-verbs-and-plan-merge-cadence.md` ("invoke keel verbs via the Skill tool rather than
re-enacting them from source"), because reading `SKILL.md` directly becomes the only way to
dogfood a just-built verb. **This harvest run is itself an instance** — the `harvest` skill
refused to load via the Skill tool, and the verb was executed by reading its source.

## D2. Environment target is never classified — local vs hosted, and which port
**Status: new.** **Grain: `spec-change`** (`provision` + `references/profile-interface.md`).
Jarvis `4020ae55`; BidLevel `08f89ecd`; Parcelint `f81a27fc`.

Four instances of one missing capability — *classify and validate the environment's target before
anything runs against it*:

- Copied `.env.local` carrying hosted keys against a local stack (Jarvis).
- Port collisions across concurrent projects, escalated by the user himself: *"we have to select
  a new standard port for this project we will use moving forward to not clash with other
  projects."*
- Build-time env inlining that ate a whole verify session and left **four chore specs** behind
  (BidLevel `dedicated-dev-port.md`, `local-auth-redirect-port.md`, `prod-walk-env.md`,
  `middleware-to-proxy.md`). The user's note — *"we keep running into"* — marks it recurrent.
- Un-provisioned-vs-broken confusion at kickoff: *"we not yet run provision so that is why
  right?"*

**Jarvis's port rule is worth promoting verbatim into the stack profile**: derive the port from
the project name (it landed on 24685). Two properties beat any specific number — it is
*effectively random* (no `/etc/services` claim) and *derivable* (any project applying the rule
gets its own port, with no registry, and it never drifts between machines or checkouts) — and the
band sits **below 49152**, where macOS begins allocating ephemeral outbound ports, so the OS can
never transiently take it. Paired note: changing it requires updating **both** `supabase/config.toml`
(needs a stack restart) and the hosted dashboard if `.env.local` points there.

Also: test setup should never depend on `.env.local`. Jarvis's committed pattern — read
`supabase status`, map names to the architecture contract, swallow non-zero if no stack is up —
belongs in the profile.

## D3. Mid-run human-credential stalls, and the one project that pre-drained them
**Status: new.** **Grain: `spec-change`** (`provision`).
Jarvis `4020ae55`/`97917aba`; Parcelint `f81a27fc`; BidLevel `2eb45dd6`; Relay `310d7574`.

Two halts in one day in Jarvis (API credits exhausted mid-verification; Docs/Sheets OAuth
connections missing) against **one project that drained it at kickoff**: Parcelint provisioned a
**hard-dollar-capped provider key** and allowlisted the spending command, so unattended runs sit
inside a bounded worst case. Promote that from a recorded learning to a `provision` step.

Adjacent provisioning friction, all pointing the same way:

- **Third-party consoles with no CLI path** (BidLevel/Google OAuth): the longest human-friction
  stretch in that project's week — *"give me clear step by-step instructions"*, *"I just completed
  number one. Now what?"*, *"is there a google cloud console api, mcp, or cli that you can use?"*
  Handled as improvised back-and-forth rather than a resumable checklist. An agent-supplied
  redirect URI was also **wrong** (app port vs local Supabase Auth port), self-flagged as *"silent
  until a live OAuth consent round-trip fails."*
- **OAuth-app ownership ambiguity** (Jarvis/Composio): all five auth configs returned
  `managed=true` — Composio's own Google app — so the user was about to edit the wrong console.
  Provision should report *who owns the OAuth app* per connected service.
- **GitHub branch-protection API** (Parcelint): the targeted endpoint 404s; the full protection
  `PUT` **replaces the whole object**, so every existing setting had to be rebuilt by hand.
  Provision should use read-modify-write by default.
- Relay's four-service cascade (macOS runner → EAS → cloud Supabase → auth re-gate → stable URL
  → a Vercel Custom Environment consuming the only Pro slot), each link discovered after
  committing to the last. One profile fact — *where does the app under test run relative to its
  backend?* — asked at the deferral stage would have surfaced the chain.

`provision` should be **re-invocable mid-project**, emit human-console steps as a persistent
checkable list, and carry connector/MCP authorization in its preflight.

# The un-keeled control case

`test landing page` is the only non-keel-managed project in the corpus, and it is the argument for
keel stated negatively. Nothing exotic went wrong — **the ordinary things did**:

- **No version control at all.** A multi-hour build with a mid-session direction pivot and
  repeated "make it longer, more animation" expansions ran with no branch, no PR, no revert point.
  The agent's own note: *"even though there's no git repo, I'll add a .gitignore anyway."*
- **An API key wired into client-side code by design** — a `VITE_`-prefixed var ships to the
  browser. The agent flagged it as demo-only and built the feature on it anyway. The `.gitignore`
  was created *before the repo existed*, so it protected nothing.
- **Model identifiers swapped with no gate** — a live-key probe was run first (good instinct), but
  the swap landed with no commit and no test, so nothing pins that the app still works.
- **Requirements grew by verbal accretion with no spec**, so nothing could be verified and nothing
  could be declared finished.

Every one of these is gated in the keel-managed projects, and in Jarvis each equivalent defect was
caught *because a done-condition named it*.

# Wins observed — keep these

- **`review-feature` is the most consistently valuable verb in the corpus.** It caught what
  autonomous verification structurally cannot, in three separate projects: BidLevel's cramped
  inline-edit column and two-axis scroll issues; Jarvis's `"sheetof"` JSX-transform bug (confirmed
  in the compiled chunk) and a 533px-at-390px mobile clip (found by DOM ancestor walk, fixed with
  `min-w-0`, right edge 509 → 366); Parcelint's observation that a twilight map direction is wrong
